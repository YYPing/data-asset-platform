## 第一章 项目概述

### 1.1 项目背景

为支持政府数据资产登记试点方案，建设独立的数据资产全流程管理平台。平台需符合《企业数据资源相关会计处理暂行规定》、《数据安全法》、《个人信息保护法》等法律法规要求，实现数据资产从登记到退出的全生命周期管理。

### 1.2 项目目标

1. 支持数据持有方自主管理数据资产
2. 支持登记中心监控全市数据资产情况
3. 支持第三方评估机构参与评估工作
4. 实现10阶段全生命周期管理
5. 实现全流程材料保存和防篡改
6. 提供大屏展示和决策支持
7. 满足等保三级安全要求

### 1.3 项目范围

**核心功能**：
- 数据资产登记与管理
- 全流程材料管理（上传、版本控制、哈希存证）
- 合规评估与价值评估
- 权属确认与确权登记
- 入账管理（无形资产/存货判断）
- 运营管理与变更管理
- 监管报送与退出注销
- 大屏展示与统计分析
- 审计日志与安全管理

**不在范围内**：
- 与外部平台API集成（预留接口）
- 区块链存证（二期规划）
- 移动端APP（响应式设计覆盖）

### 1.4 项目约束

- **用户规模**: 试点阶段<100并发用户
- **部署环境**: 政府内网，需支持离线部署
- **安全要求**: 等保三级
- **开发周期**: 6个月（MVP三阶段）
- **团队规模**: 5-6人

## 第三章 业务流程设计

### 3.1 完整生命周期（10阶段）

> **V2.0核心变更**: 从7阶段扩展为10阶段完整生命周期

```
① 登记 → ② 合规评估 → ③ 权属确认 → ④ 确权登记 → ⑤ 价值评估
    ↓                                                    ↓
⑩ 退出/注销 ← ⑨ 监管 ← ⑧ 变更管理 ← ⑦ 运营 ← ⑥ 入账

横向流程（可在任意阶段触发）：
  - 争议处理（申诉→复核→仲裁）
  - 紧急冻结（安全事件时冻结资产）
```

**阶段说明**：

**① 登记阶段**（数据持有方操作）：
- 上传数据情况说明（数据规模、格式、来源、更新频率等）
- 上传合规报告（数据安全评估、个人信息保护影响评估等）
- 上传数据质量报告（完整性、准确性、时效性评估）
- 保存外部导入的**登记证书**（PDF/图片，系统记录证书编号和有效期）
- 系统自动计算所有上传材料的**SHA256哈希值**并存证
- 记录证书有效期，到期前自动提醒续期

**② 合规评估**：基于登记材料进行法规合规性审查
**③ 权属确认**：确认数据资产的权属归属（先确权再估值）
**④ 确权登记**：对接登记机构，完成正式确权登记，公示
**⑤ 价值评估**：在权属明确后进行价值评估（成本法/市场法/收益法）
**⑥ 入账**：根据评估结果判断入表类型（无形资产/存货），完成会计处理
**⑦ 运营**：资产授权使用、收益管理、质量监控
**⑧ 变更管理**：权属变更、价值重估、用途变更等
**⑨ 监管**：定期报送、现场检查、整改跟踪
**⑩ 退出/注销**：资产处置、注销登记、归档保存

### 3.2 流程状态机

```
[草稿] → [预审中] → [已提交] → [审核中] → [补正] → [已通过] → [公示中] → [已生效]
  ↑         ↓           ↓          ↓         ↑        ↓
  └── [已驳回] ←────────┴──────────┘         │    [已归档]
                                              │        ↓
                                          [变更中]  [已注销]
```

**状态说明**：
| 状态 | 说明 | 可执行操作 |
|------|------|-----------|
| 草稿 | 持有方填写中，未提交 | 编辑、删除、提交预审 |
| 预审中 | 系统自动检查材料完整性 | 自动通过/退回补充 |
| 已提交 | 正式提交到登记中心 | 等待分配审核人 |
| 审核中 | 登记中心审核 | 通过、驳回、要求补正 |
| 补正 | 持有方补充材料 | 编辑、重新提交 |
| 已通过 | 审核通过 | 进入下一阶段 |
| 公示中 | 确权登记公示期 | 等待公示期满 |
| 已生效 | 正式生效 | 运营、变更、退出 |
| 已驳回 | 审核不通过 | 修改后重新提交或放弃 |
| 变更中 | 资产信息变更审批中 | 等待审批 |
| 已归档 | 资产归档保存 | 只读查询 |
| 已注销 | 资产注销退出 | 只读查询 |

### 3.3 审批流程设计

**普通资产审批**（2级）：
```
持有方提交 → 登记中心初审 → 登记中心终审 → 通过/驳回
```

**重大资产审批**（3级，价值>100万）：
```
持有方提交 → 登记中心初审 → 第三方评估 → 登记中心终审 → 通过/驳回
```

**驳回回退机制**：
- 驳回时必须指定回退环节和原因
- 持有方可在线修改后重新提交
- 同一环节驳回超过3次，转人工专项处理
- 驳回历史完整记录

**超时处理规则**：
| 环节 | 处理时限 | 超时动作 |
|------|---------|---------|
| 预审 | 1个工作日 | 自动通过（材料完整时） |
| 初审 | 5个工作日 | 提醒审核人+上级 |
| 第三方评估 | 15个工作日 | 提醒评估机构+登记中心 |
| 终审 | 3个工作日 | 提醒审核人+上级 |
| 补正 | 10个工作日 | 提醒持有方，超30日自动关闭 |
| 公示期 | 5个工作日 | 自动进入下一阶段 |

### 3.4 并行审批支持

合规评估和价值评估可并行进行：
```
                    ┌→ [合规评估] ─┐
[权属确认通过] ──→ │              ├→ [综合审查] → [确权登记]
                    └→ [价值评估] ─┘
```

## 第四章 功能模块设计

### 4.1 用户管理模块

**功能清单**：
1. 用户注册与登录（用户名+密码）
2. JWT令牌认证（Access Token 15分钟 + Refresh Token 7天）
3. 密码策略（长度≥8、含大小写+数字+特殊字符、90天过期）
4. 登录失败锁定（5次失败锁定30分钟）
5. 组织机构管理（树形结构）
6. 个人信息管理

### 4.2 数据资产管理模块

**功能清单**：
1. 资产登记（基本信息、数据目录、来源说明）
2. 资产信息维护（编辑、版本控制）
3. 资产状态跟踪（10阶段状态流转）
4. 资产查询（条件筛选、全文搜索）
5. 资产统计（按类型、状态、组织统计）
6. 资产导入导出（Excel模板批量导入）
7. 资产复制创建（相似资产快速创建）
8. 重复登记检查（防止同一资产重复登记）

### 4.3 材料管理模块

**功能清单**：
1. 材料上传（支持拖拽、批量上传、格式白名单）
2. 材料分类（按阶段自动分类）
3. 标准化材料清单（各阶段必需材料模板）
4. 版本控制（MinIO Bucket Versioning）
5. 哈希存证（SHA256校验，防篡改）
6. 在线预览（图片、PDF）
7. 审核流程（待审核→审核中→通过/驳回）
8. 材料完整性自动检查

**各阶段材料清单**：
| 阶段 | 必需材料 |
|------|---------|
| 登记 | 数据情况说明书、数据质量报告、合规报告、外部登记证书（导入）、营业执照/组织代码 |
| 合规评估 | 数据情况说明书、数据质量报告、数据安全评估报告、个人信息保护影响评估、合规自查表 |
| 权属确认 | 数据情况说明书、数据来源协议、采集授权书、加工记录、权属声明 |
| 确权登记 | 数据情况说明书、登记申请表、权属证明材料汇总、确权登记证书（系统生成/导入） |
| 价值评估 | 数据情况说明书、数据质量报告、评估委托书、第三方评估报告、评估方法说明 |
| 入账 | 数据情况说明书、数据质量报告、会计处理方案、入表类型判断依据、财务审批单 |

**登记证书管理**：
- 支持外部导入登记证书（PDF/JPG/PNG）
- 系统自动记录证书编号、颁发机构、颁发日期、有效期
- 证书文件自动计算SHA256哈希值存证（防篡改）
- 有效期到期前30天/15天/7天自动提醒续期
- 证书状态管理：有效/即将过期/已过期/已撤销

### 4.4 评估模块

**合规评估**：
1. 法规检查清单（《数据安全法》《个人信息保护法》《网络安全法》）

### 4.5 工作流模块

**功能清单**：

2. 审批任务分配（自动/手动）

**交互能力**：

## 第八章 安全设计

> **V2.0核心变更**: 从笼统描述到完整安全方案

### 8.1 身份认证

1. **JWT认证**: Access Token(15分钟) + Refresh Token(7天)
2. **Token黑名单**: 登出时Token加入Redis黑名单
3. **密码策略**: 长度≥8，含大小写+数字+特殊字符，90天强制更换
4. **登录锁定**: 连续5次失败锁定30分钟
5. **会话管理**: 单设备登录，新登录踢出旧会话

### 8.2 权限控制

1. **RBAC模型**: 用户→角色→权限→资源
2. **数据权限**: 自动附加组织过滤（持有方只看本组织数据）
3. **字段脱敏**: 手机号显示为138****1234，身份证显示为110***********1234
4. **操作权限**: 每个API接口绑定权限码，中间件统一校验

### 8.3 数据安全

1. **传输加密**: 全站HTTPS（TLS 1.2+）
2. **密码存储**: bcrypt哈希（cost=12）
3. **文件防篡改**: 上传时计算SHA256哈希，下载时校验
4. **数据备份**: 每日全量备份（凌晨2点），保留30天
5. **敏感数据**: 数据库字段级加密（AES-256）

### 8.4 审计日志

1. **全操作记录**: 谁、何时、做了什么、结果如何、IP地址
2. **日志防篡改**: 写入后不可修改，按月分区归档
3. **保留期限**: 最少3年
4. **查询能力**: 按用户、时间、操作类型、资源筛选

### 8.5 应用安全

1. **输入验证**: Pydantic模型强制校验所有输入
2. **SQL注入防护**: SQLAlchemy参数化查询
3. **XSS防护**: 前端输出转义，CSP头
4. **CSRF防护**: SameSite Cookie + Token验证
5. **文件上传安全**: 格式白名单（pdf/jpg/png/xlsx/docx），大小限制100MB
6. **API限流**: 100次/分钟/IP（Redis令牌桶）

### 8.6 等保三级要求

| 要求项 | 实现方案 |
|--------|---------|
| 身份鉴别 | JWT+密码策略+登录锁定 |
| 访问控制 | RBAC+数据权限隔离 |
| 安全审计 | 全操作审计日志，保留3年 |
| 数据完整性 | 文件SHA256校验 |
| 数据保密性 | HTTPS+字段加密+脱敏 |
| 备份恢复 | 每日备份，季度恢复演练 |

### 10.3 Nginx负载均衡配置

> **评审修正C8**: 补充Nginx upstream配置和健康检查

```nginx
upstream app_backend {
    server app-a:8000 weight=1 max_fails=3 fail_timeout=30s;
    server app-b:8000 weight=1 max_fails=3 fail_timeout=30s backup;
    # 健康检查：3次失败后摘除，30秒后重试
}

server {
    listen 443 ssl;
    server_name data-asset.example.com;

    ssl_certificate     /etc/nginx/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/ssl/key.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;

    # 静态资源
    location / {
        root /usr/share/nginx/html;
        try_files $uri $uri/ /index.html;
        expires 7d;
    }

    # API代理
    location /api/ {
        proxy_pass http://app_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_connect_timeout 10s;
        proxy_read_timeout 60s;
        
        # 文件上传大小限制
        client_max_body_size 100M;
        client_body_timeout 300s;
    }

    # 健康检查端点
    location /health {
        proxy_pass http://app_backend/api/v1/health;
        access_log off;
    }
}
```

### 10.5 备份恢复策略

### 10.6 监控告警方案

> **评审修正C6**: 补充完整监控告警方案

**监控层次**：
```
1. 基础设施监控（Prometheus + node_exporter）:
   - CPU使用率 > 80% → 黄色告警
   - 内存使用率 > 85% → 黄色告警
   - 磁盘使用率 > 90% → 红色告警
   - 网络异常 → 红色告警

   - API响应时间P95 > 500ms → 黄色告警
   - API错误率 > 1% → 红色告警
   - 应用进程不存在 → 红色告警
   - 健康检查失败 → 红色告警

3. 数据库监控（Prometheus + pg_exporter）:
   - 连接数 > 40（max=50的80%）→ 黄色告警
   - 慢查询 > 1s → 记录日志
   - 主从复制延迟 > 10s → 红色告警
   - 磁盘空间 < 20% → 红色告警

4. 业务监控（自定义指标）:
   - 日登记量、审批时长、驳回率
   - 证书即将过期数量
   - 待办积压数量
```

**告警通道**：

|------|------|---------|

| 黄色（警告） | 邮件 + 企业微信 | 1小时内响应 |
| 蓝色（信息） | 邮件 | 下一工作日处理 |

**日志聚合**（Loki + Grafana）：
- 应用日志：JSON格式，包含request_id、user_id、耗时

```
发布流程：

2. 先更新app-b（备机），验证健康检查通过
3. Nginx切换流量到app-b，更新app-a

3. 数据库回滚：提前备份，执行回滚SQL脚本
4. 回滚完成后通知相关人员
```
